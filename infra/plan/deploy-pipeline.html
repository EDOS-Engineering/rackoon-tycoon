3 · Deploy Pipeline

GitHub Actions verifies, then triggers Forge. Two mechanisms — webhook (recommended) or SSH/rsync.

Shape of the pipeline

The workflow .github/workflows/deploy.yml (in this branch) runs on push to main and on manual workflow_dispatch:

  1. verify job — checks out the repo, sets up Node, runs node tooling/headless.mjs (the dev-only seedable sim harness). Optionally runs the Playwright smoke test. This is a gate: a failing sim blocks the deploy.
  2. deploy job (needs: verify) — POSTs the Forge deploy trigger URL with curl, then polls/health-checks https://rackoon.cloud for a 200 + a known string.

Mechanism A — Forge Deploy Webhook (recommended)

Forge exposes a per-site Deploy Trigger URL. POSTing to it runs the site's deploy script on the droplet. Forge holds the SSH/git credentials; GitHub never touches the server directly.

How to get the URL in Forge

  1. Forge → site rackoon.cloud → Deployment tab.
  2. Find Deploy Trigger URL (a.k.a. "Deployment Webhook"). It looks like https://forge.laravel.com/servers/<id>/sites/<id>/deploy/http?token=<secret>.
  3. Copy it. Leave Forge's own Quick Deploy (auto-deploy on git push) OFF — we want Actions to gate on verify first, not deploy on raw push.

Required GitHub repository secret

SecretValue
FORGE_DEPLOY_WEBHOOK_URLThe full Deploy Trigger URL above (includes its own ?token=). Treat as a secret — it can trigger deploys.

Set it in GitHub → repo → Settings → Secrets and variables → Actions → New repository secret.

The deploy step is just:

curl -fsS -X POST "$FORGE_DEPLOY_WEBHOOK_URL"
Why recommended: no SSH keys to manage in GitHub, no inbound SSH exposure, Forge keeps the deploy logic and credentials. One secret. Smallest blast radius.

Mechanism B — SSH / rsync (alternative)

GitHub Actions SSHes into the droplet and either runs git pull or rsyncs the files. Use only if you want GitHub to drive the server directly (e.g. to bypass Forge for deploys). Shipped as .github/workflows/deploy-ssh.yml, disabled by default.

Required GitHub repository secrets

SecretValue / how to get it
SSH_HOSTDroplet static IP (<DROPLET_STATIC_IP>) or rackoon.cloud.
SSH_USERThe Forge deploy user, normally forge.
SSH_KEYA dedicated deploy private key (PEM). Generate ssh-keygen -t ed25519 -C "gha-deploy", add the public half to the droplet (Forge → server → SSH Keys, or ~forge/.ssh/authorized_keys); put the private half here. Do not reuse a personal key.
DEPLOY_PATHServer deploy dir, e.g. /home/forge/rackoon.cloud.
SSH_KNOWN_HOSTS(recommended) Output of ssh-keyscan <DROPLET_STATIC_IP>, to pin the host key instead of blindly trusting it.

The deploy step (git-pull flavour) is essentially:

ssh "$SSH_USER@$SSH_HOST" "cd $DEPLOY_PATH && git fetch origin main && git reset --hard origin/main"

An rsync flavour (rsync -az --delete ./ "$SSH_USER@$SSH_HOST:$DEPLOY_PATH/" with tooling/ and .git/ excluded) is shown commented in the workflow file.

Which to use

Use Mechanism A (Forge webhook). It is one secret, no key management, and keeps Forge as the single source of deploy truth. Keep B as the documented fallback (disabled workflow) in case you ever move off Forge-driven deploys.

Health check (both mechanisms)

After triggering, Actions confirms the live site:

# Retry a few times to allow the git pull on the droplet to finish.
for i in $(seq 1 10); do
  body=$(curl -fsS --max-time 10 https://rackoon.cloud/game/game.html || true)
  echo "$body" | grep -q "Rackoon" && { echo "healthy"; exit 0; }
  sleep 6
done
echo "health check failed"; exit 1
