2 · Forge Setup

Create a Static HTML site for rackoon.cloud on the existing server, connect the repo, set the web directory, enable SSL, and wire the deploy script. no build step

URL layout — the decision

Recommended layout:
  • https://rackoon.cloud/ → the game (redirect / to /game/game.html).
  • https://rackoon.cloud/study → the SAA-C03 study guide (the repo's root index.html).
  • https://rackoon.cloud/game/… → game assets/modules, served as-is.

The game is the headline experience, so it owns /. The study guide gets a clean /study path. Because the repo has no build step, we serve the repository root directly and shape the URLs entirely in nginx — no file moves, no copies.

⚠ Note for the human (do not let CI fix this). game/game.html has a hard-coded link back to the study guide as ../index.html. Served at /game/game.html, that link resolves to /index.html — which (with the redirect below) bounces to the game, not the study guide. Two clean options, your call:
  • (A) nginx-only, no code change (used below): keep /index.html served literally as the study guide and only redirect the bare / to the game. Then ../index.html still lands on the study guide, and /study is an extra alias. Simplest; ships today.
  • (B) one-line code change (cleaner, do later): change the link in game/game.html from ../index.html to /study, then redirect /index.html → /study too. This is a tweak to an existing file, so it is noted here, not done by this plan.
The nginx block below implements (A) so it works with the repo exactly as-is.

Step-by-step in the Forge UI

  1. Open Forge → your existing server. Under Sites, click New Site.
  2. Root Domain: rackoon.cloud. Project Type: Static HTML / Nuxt / etc. (the "Static HTML" option — no PHP runtime needed). Add www.rackoon.cloud under Aliases. Create the site.
  3. On the new site: Git Repository → Install Repository. Provider GitHub, repository EDOS-Engineering/rackoon-tycoon, branch main. Uncheck "Install Composer Dependencies" (there are none).
  4. Web directory. The repo serves from its own root (the game lives at game/, the study guide at the root index.html), so set the site's Web Directory to / (i.e. the repo root, not /public). The deploy path on disk is typically /home/forge/rackoon.cloud.
  5. Deploy script. Forge → site → Deployment tab → edit the deploy script to the no-build version (below). Save.
  6. nginx config. Forge → site → ⋯ menu → Edit Nginx Configuration. Replace the location / block (and add the helpers) with the config below. Save — Forge validates and reloads nginx.
  7. SSL. Forge → site → SSL → Let's Encrypt → include rackoon.cloud and www.rackoon.cloud → Obtain Certificate. (DNS must already resolve to the droplet — see DNS page.) Then toggle Force HTTPS on.
  8. First deploy. Click Deploy Now once to pull the repo and confirm the site renders. After this, deploys come from GitHub Actions (see next page).

Deploy script (no build)

Forge runs this on the droplet on each deploy. Pure git pull — no npm, no bundler.

cd /home/forge/rackoon.cloud

# Fetch and hard-reset to the deployed branch so the working tree always
# matches origin/main exactly (avoids "local changes" stalls on a static repo).
git fetch origin main
git reset --hard origin/main

# No build step: zero runtime deps, no bundler. Static files are served as-is.
# nginx serves the new files immediately; no reload needed for content changes.
echo "Deployed rackoon.cloud @ $(git rev-parse --short HEAD)"

Forge auto-wraps this so a deploy already in progress won't overlap. Keep the script idempotent (the reset --hard guarantees that).

nginx — URL layout + correct MIME types

Paste into the site's nginx config, replacing the default location /. root is the deploy path; Forge fills the rest of the server block (TLS, logs).

# --- Rackoon Tycoon static layout ---------------------------------
# root is the repo root on disk (Forge sets this in the server block):
#   root /home/forge/rackoon.cloud;

# Make sure JS / ES modules ship as text/javascript (browsers refuse to
# execute modules served as the wrong MIME type). Forge's mime.types maps
# .js already on modern nginx; .mjs may not be — force both to be safe.
types {
    text/javascript  js mjs;
}

# Bare site root → the game (headline experience).
location = / {
    return 302 /game/game.html;
}

# Study guide at a clean path. alias maps /study and /study/ to the repo
# root index.html (the SAA-C03 guide). try_files keeps it a single page.
location = /study {
    alias /home/forge/rackoon.cloud/index.html;
    default_type text/html;
}
location = /study/ {
    alias /home/forge/rackoon.cloud/index.html;
    default_type text/html;
}

# Everything else: serve the static file if it exists, else 404.
# This serves /game/game.html, /game/src/*.js, /index.html (the study
# guide, kept literal so the game's ../index.html back-link still works),
# images, etc. No SPA fallback — every URL is a real file.
location / {
    try_files $uri $uri/ =404;
}

# Long cache for fingerprint-free static assets is handled per-type in the
# Security & Rollback page (HTML no-cache, JS/img long-cache). Add those
# location blocks here too.
# ------------------------------------------------------------------
Why this works with no build: the repo is already a valid static tree. nginx serves it verbatim; the only shaping is one redirect (/ → game), two aliases (/study), and a MIME override for .mjs. Nothing is compiled, copied, or rewritten on disk.
