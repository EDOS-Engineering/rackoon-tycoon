4 · Security & Rollback

Least-privilege creds, no secrets in the repo, HTTPS/HSTS, sane cache headers, and a fast rollback.

Credentials & secrets

  • No secrets in the repo — ever. Everything sensitive lives in GitHub Actions secrets only (FORGE_DEPLOY_WEBHOOK_URL for the recommended path). The repo stays a clean static tree. A pre-merge git grep for token/key patterns is a cheap guard.
  • Least privilege. The Forge deploy webhook can only trigger this site's deploy script — it can't read other sites or run arbitrary commands. Prefer it over SSH for exactly this reason.
  • If using SSH (Mechanism B): use a dedicated ed25519 deploy key (not a personal key), scoped to the forge user; consider an authorized_keys command= / restrict prefix to limit it to the deploy command. Pin the host key via SSH_KNOWN_HOSTS. Rotate if leaked.
  • Rotation. Both the Forge webhook token and the SSH key can be regenerated in Forge and re-pasted into GitHub secrets at any time. No repo change needed.
  • Firewall. Keep the droplet firewall (Forge/UFW) allowing only 80/443 (+ 22 from trusted sources). The webhook path needs no inbound SSH from GitHub at all.

HTTPS / HSTS

  • Force HTTPS in Forge (redirects :80 → :443). Let's Encrypt cert covers rackoon.cloud + www; Forge auto-renews it.
  • HSTS — add the header once you're confident HTTPS is stable (so you don't lock users out during cutover):
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

Start with a shorter max-age (e.g. 300) for the first day, then raise to a year. Only add preload if you intend to submit to the HSTS preload list (hard to undo).

Cache headers

The trap with a no-build static site: assets have no content hash in their names, so aggressive caching can serve stale JS after a deploy. Strategy: long-cache the assets, never-cache the HTML entry points so a refresh always re-checks them and they re-fetch any changed module.

# HTML entry points: always revalidate so updates show immediately.
location ~* \.html$ {
    add_header Cache-Control "no-cache, must-revalidate" always;
}

# JS / ES modules, images, audio: long cache. (No filename hashes, so we
# still revalidate the HTML above; if you change a module's *contents* and
# need an instant bust, append a ?v=NN query in the importing HTML.)
location ~* \.(mjs|js|png|jpg|jpeg|svg|webp|gif|woff2?|mp3|ogg)$ {
    add_header Cache-Control "public, max-age=604800" always;
}
Caveat: ES modules import each other by bare path, so a long-cached .js can pin an old module even after the entry HTML updates. For this static, no-hash repo a 7-day asset cache is a safe middle ground; for guaranteed-instant updates, drop module assets to no-cache too (the files are small) or add ?v= query strings when you bump them.

Rollback

Forge deploys are just git reset --hard on the droplet, so rollback = redeploy a previous commit. Fast and clean:

  1. Revert in git (preferred): git revert <bad-sha> on main and push. The pipeline re-runs verify + deploys the reverted tree. Full audit trail.
  2. Hot rollback on the droplet (fastest): SSH in and check out the last-good commit: 
    cd /home/forge/rackoon.cloud
git fetch origin
git checkout <last-good-sha>   # detached HEAD, serves immediately
# then click Deploy in Forge to return to origin/main once fixed
  3. Forge redeploy: Forge keeps deploy history; trigger a deploy after pointing main at the good commit.

Because there's no build artifact and no database, rollback is instantaneous — nginx serves whatever files are on disk.

Post-deploy smoke test

The pipeline's last step (and a good manual check) — confirm a 200 and a known string on both surfaces:

# Game entry (root redirects here)
curl -fsS -o /dev/null -w "%{http_code}\n" https://rackoon.cloud/game/game.html   # expect 200
curl -fsS https://rackoon.cloud/game/game.html | grep -q "Rackoon Tycoon"          # known string

# Study guide
curl -fsS -o /dev/null -w "%{http_code}\n" https://rackoon.cloud/study             # expect 200

# Root redirect
curl -fsS -o /dev/null -w "%{http_code} %{redirect_url}\n" https://rackoon.cloud/  # expect 302 -> /game/game.html

# A module serves as JavaScript (catches the .mjs MIME trap)
curl -fsSI https://rackoon.cloud/game/src/main.js | grep -i "content-type"         # expect text/javascript
